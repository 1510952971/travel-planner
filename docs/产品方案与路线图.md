## 1. 产品定位

帮助用户**快速生成、多人协作、地图可视化、预算 AA** 的旅行规划工具。  
第一原则：**先打透核心痛点，再扩平台与外部 API**。

---

## 2. MVP 核心功能（与方案对齐）

| # | 能力 | 用户价值 | 阶段 |
|---|------|----------|------|
| 1 | **智能行程创建** | 输入目的地 / 天数 / 偏好 → 基础日程 | **P0 当前** |
| 2 | **清单与备忘** | 行前准备、证件衣物药品 | **P0 已有 + 加强** |

| 8 | **AI 一句话生成** | 对接大模型 | P2 |
| 9 | **小票识别** | OCR 记账 | P3 |

## 3. 分阶段路线图

### P0 — 本机 Web MVP（现在）

**目标：** 单机可用的「行程大脑」，验证产品逻辑。

| 模块 | 实现方式 |
|------|----------|
| 前端 | 静态 HTML/CSS/JS（已有） |
| 存储 | localStorage + JSON 导入导出 |
| 智能行程 | 规则模板引擎（按偏好生成），预留 AI 接口 |
| 预算 AA | 多人分摊结算表（本地） |
| 平台 | 浏览器（Windows 可直接打开） |

**明确不做（P0）：** 真账号体系、实时协作、真实地图 SDK、上架应用商店。

### P1 — 地图与真实数据

- 高德 / 百度（国内）或 Google Maps（全球）
- 日程点在地图上打点、折线连日路线
- 一键唤起导航 App（深链）
- 和风天气 / OpenWeather 展示目的地天气

### P2 — 后端 + 协同 + 多端

| 模块 | 建议 |
|------|------|
| 后端 | Python FastAPI 或 Node NestJS |
| 库 | PostgreSQL（用户/行程元数据）+ MongoDB 或 JSONB（日程结构） |
| 协同 | WebSocket / CRDT（可先「共享只读链接 + 邀请编辑」） |
| App | 验证 Web 后再 Flutter 或微信小程序 |
| AI | 对接 Gemini / 其它 LLM，一句话生成行程 |

### P3 — 差异化

- 离线包（Service Worker + 瓦片缓存策略）
- TSP 简化排序
- 小票 OCR
- 多币种汇率

## 4. 技术栈决策（当前建议）

| 问题 | 建议 | 原因 |
|------|------|------|
| 先做哪个端？ | **Web（本机浏览器）→ 再小程序/App** | 最快验证；你已有 `travel-planner` 骨架 |
| 是否一上来 Flutter？ | **否** | 协作/地图/API 未稳时，改需求成本高 |
| 后端何时上？ | 要做**登录协同**或**多设备同步**时再上 | 避免过早基建 |
| 地图国内/全球？ | 主国内用 **高德**；出境再补 Google | 合规与密钥成本 |
